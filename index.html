Module 1

Blockchain basics

A blockchain is a shared record book that no single person controls. Before diving into the cryptography, let’s understand the problem it solves.

The problem: trusting a record keeper

Banks, land registries and payment networks all work the same way: one trusted party keeps the official record. That works — until the record keeper makes a mistake, gets hacked, censors you, or simply isn’t trusted by everyone involved.

A blockchain flips the model: everyone keeps a full copy of the ledger, new entries are only added when the network agrees on them, and old entries are mathematically protected from editing.

Four properties to remember

Distributed
Thousands of computers (nodes) each hold the complete ledger.
Append-only
You can add new records, but never edit or delete old ones.
Tamper-evident
If anyone alters history, every other copy immediately exposes the change.
Consensus-driven
Rules — not a boss — decide which new entries are accepted.

Try it: the network rejects a cheat

Three nodes hold identical copies of a tiny ledger. Tamper with one and see what the majority does.

Node B can scribble whatever it wants in its own copy — but A and C still agree on the truth, so the network ignores B. To rewrite history you’d have to out-vote the entire honest network.

Where does cryptocurrency fit in?

A cryptocurrency is simply what the ledger tracks: entries like “Alice pays Bob 5 coins”. Because the ledger can’t be forged or rewritten, those entries behave like money — scarce, transferable, and not dependent on any bank. Bitcoin (2009) was the first system to make this work; thousands of others followed.

Forty years of failed digital cash — then Bitcoin

Digital money is an old dream with one stubborn bug: a digital coin is just data, and data can be copied. Every attempt before Bitcoin patched the copying problem the same way — put one trusted company in charge — which simply recreated a bank.

  • 1982 — Lamport, Shostak & Pease formalize the Byzantine generals problem: how can independent parties agree when some of them may lie? Every blockchain is an answer to this question.
  • 1989 — David Chaum’s DigiCash issues cryptographic e-cash — run by a company, which went bankrupt in 1998. Centralization was the single point of failure.
  • 1997 — Adam Back’s hashcash makes email senders burn CPU time to deter spam: proof of work is born (you’ll meet it in Module 4).
  • 1998 — Wei Dai’s b-money and Nick Szabo’s bit gold sketch decentralized money, but neither has a working consensus mechanism.
  • October 2008 — the pseudonymous Satoshi Nakamoto publishes a nine-page whitepaper wiring all of these pieces together.
  • 3 January 2009 — the Bitcoin genesis block is mined, with a newspaper headline embedded in it forever: “Chancellor on brink of second bailout for banks.”

Notice what Bitcoin didn’t invent: hashes, signatures and proof of work all existed. The breakthrough — now called Nakamoto consensus — was wiring them together so that thousands of strangers converge on one history with nobody in charge.

Do you actually need a blockchain?

Decentralization is bought, not free. Every node stores and re-verifies everything, so a base-layer blockchain is a spectacularly slow database: Bitcoin processes ~7 transactions per second and Ethereum ~15–30, while one ordinary database server handles tens of thousands.

A useful test — a blockchain earns that cost only when all three hold:

  1. Several parties need to write to the same records…
  2. …and they don’t fully trust each other (or any single operator)…
  3. …and no acceptable middleman exists — too expensive, censoring, or itself untrusted.

If a trusted operator exists, a regular database is simpler, faster and cheaper. Most “enterprise blockchain” projects of the 2010s quietly became databases for exactly this reason.

The blockchain trilemma

Three properties everyone wants — and a base layer realistically maximizes two:

Decentralized
Anyone can run a node and verify the rules — no gatekeepers.
Secure
Rewriting history or censoring users is economically irrational.
Scalable
High throughput, low fees, fast confirmation.

The trade-offs are mechanical, not accidental: bigger or faster blocks mean fewer people can afford to run a node (less decentralized); a small hand-picked validator set is fast but reintroduces trust. Bitcoin and Ethereum deliberately sacrifice base-layer speed, then push scale to layer 2 networks that inherit their security — Module 7 shows how.

Module 2

Hashing — the fingerprint machine

Every security promise a blockchain makes traces back to one tool: the cryptographic hash function. Master this and the rest of the course clicks into place.

What a hash function does

A hash function takes any input — a letter, a novel, a 4 GB movie — and produces a fixed-size fingerprint. Bitcoin and this course use SHA-256, which always outputs 256 bits (written as 64 hex characters).

  • Deterministic — same input, same hash. Always.
  • One-way — there is no way to compute the input from the hash. Guessing is the only option.
  • Avalanche effect — change one character and the entire hash changes beyond recognition.
  • Collision-resistant — finding two inputs with the same hash is astronomically hard.

A one-character tweak flips roughly half of all 256 bits. That’s why a hash works as a tamper seal: even the tiniest edit is glaringly obvious.

Why blockchains care

Blockchains use hashes as tamper-evident seals. Hash a block’s contents and you get its fingerprint; change anything inside the block and the fingerprint no longer matches. In the next module you’ll see how chaining these fingerprints together protects an entire history.

Scaling up: Merkle trees

A real block holds thousands of transactions, but it doesn’t store one giant hash of everything. Instead it hashes transactions in pairs, then hashes those hashes in pairs, and so on, up to a single fingerprint-of-fingerprints: the Merkle root. The root goes in the block header, so the header commits to every transaction beneath it — and, as you’re about to see, proving one transaction is in the block takes only a handful of hashes.

Try it: tamper with a Merkle tree

Four transactions are hashed into a tree. Edit any one and watch the change ripple up — but only along its own path to the root.

Edit Tx C: H(C), H(CD) and the root change — H(A), H(B) and H(AB) don’t. That path is the point. To prove Tx C is in the block you hand someone just H(D) and H(AB): two hashes instead of four transactions. With a million transactions it’s only ~20 hashes — this Merkle proof is how a phone wallet verifies a payment without downloading the whole chain (Bitcoin’s whitepaper calls it SPV, simplified payment verification).

How big is 2²⁵⁶, really?

SHA-256 has 2²⁵⁶ possible outputs — about 1.2 × 10⁷⁷, within a few zeros of the number of atoms in the observable universe (~10⁸⁰). Intuition fails at this scale, so try arithmetic instead: give a billion computers a billion guesses per second each. That’s 10¹⁸ guesses per second, or ~3 × 10²⁵ per year — meaning covering even half the output space would take around 10⁵¹ years, roughly 10⁴¹ times the age of the universe.

This is why “brute-force it with a bigger computer” isn’t a threat. The bound is physical, not merely technological: just counting through 2²⁵⁶ states has a minimum thermodynamic energy cost (Landauer’s principle) far beyond the Sun’s total output. When hash functions do fall, they fall to math — a clever shortcut — not to muscle. See the next deep dive.

SHA-256’s family tree — and why old hashes get retired

SHA-256 (published 2001, part of the SHA-2 family) pushes input through 64 rounds of public, keyless bit-mixing; its security rests entirely on nobody finding structure in that mixing. Its predecessors did fall: MD5 collisions now take seconds, and SHA-1 fell in 2017 when researchers published two different PDFs with the same hash. Both falls were algorithmic breakthroughs, not bigger computers — which is why the industry retires hash functions at the first crack.

Variations you’ll meet in the wild:

  • Bitcoin applies SHA-256 twice (“SHA-256d”) when hashing blocks.
  • Ethereum uses Keccak-256 from the SHA-3 family — different internals, same fingerprint contract.
  • Addresses often layer hashes: Bitcoin runs RIPEMD-160 on top of SHA-256 (Module 5).
Hashes beyond blockchains
  • Git — every commit ID is a hash of the code plus its history; change anything and the ID changes. A repository is itself a small Merkle structure.
  • Passwords — sites store hash(password), never the password. There they use deliberately slow hashes (bcrypt, Argon2) to blunt guessing — the exact opposite of mining’s speed race.
  • Downloads & packages — published checksums let you verify a file wasn’t corrupted or tampered with in transit.
  • Deduplication & BitTorrent — identify content by what it is rather than where it lives (“content addressing”).

Module 3

Blocks & the chain

A blockchain is literally a chain of blocks — and the links are hashes. Time to break one on purpose.

Anatomy of a block

Data
The payload — typically a batch of transactions.
Previous hash
The fingerprint of the block before this one. This is the chain link.
Nonce
A number miners adjust until the block’s hash meets the difficulty rule (more in Module 4).
Hash
The fingerprint of everything above. Here, a block is valid when its hash starts with 00.

Because each block embeds the previous block’s hash, the blocks form a chain: changing any old block changes its hash, which breaks the next block’s “previous hash”, which breaks the one after that — a domino effect all the way to the tip.

Try it: break the chain, then repair it

  1. Edit the data in Block #1 (change a name or an amount).
  2. Watch it and every later block turn invalid — the dominoes fall.
  3. Press Mine on each broken block, left to right, to repair the chain.

On a real network you’d have to redo this repair work faster than thousands of honest miners extending the chain — and that’s exactly why old blocks are effectively permanent.

Inside a real Bitcoin block header

Our toy blocks hash index | prev hash | data | nonce. A real Bitcoin block header is exactly 80 bytes:

FieldSizeWhat it’s for
Version4 BSignals which protocol rules the miner follows
Previous block hash32 BThe chain link you just broke in the demo
Merkle root32 BOne hash committing to every transaction in the block (Module 2)
Timestamp4 BApproximate creation time, kept honest by consensus tolerances
Bits4 BCompact encoding of the difficulty target the hash must beat
Nonce4 BThe miner’s guess counter

Transactions live in the block body; only this 80-byte header is hashed during mining, which is why mining speed doesn’t depend on how full a block is. And because a 4-byte nonce allows only ~4.3 billion guesses — an ASIC exhausts that in well under a second — miners also tweak the timestamp and an extraNonce inside the first transaction, which changes the Merkle root and opens up a fresh 4.3 billion.

Ties, orphans and the longest-chain rule

Every so often two miners find valid blocks within seconds of each other. Both are valid; nodes near miner A hear A’s block first, nodes near B hear B’s. The tie isn’t debated — it’s raced: whichever branch receives the next block becomes the longer chain, and every node hops onto it. The losing block is orphaned (“stale”): its mining reward evaporates and its transactions return to the mempool to be confirmed again.

Two practical consequences. First, “longest” really means most accumulated work — a million easy blocks don’t outrank ten hard ones. Second, this is exactly why recipients wait for confirmations: a transaction one block deep can still land on the losing side of a tie, but each further block makes reversal exponentially less likely — hence the ~6-confirmation custom for large Bitcoin payments.

How heavy is the chain?

A Bitcoin block carries roughly 2,000–4,000 transactions every ~10 minutes, and the full chain since 2009 runs to hundreds of gigabytes — Ethereum’s state is heavier still. The ecosystem copes in three tiers: full nodes verify everything (the gold standard); pruned nodes verify everything but discard old block bodies, keeping headers and current balances; and light clients download just the 80-byte headers and check individual transactions with Merkle proofs (Module 2). The design fights hard to keep at least light verification cheap — if ordinary people can’t verify the chain, decentralization is a slogan.

Module 4

Mining & proof of work

Mining sounds mysterious, but you just did it in Module 3. It’s a guessing game — played at planetary scale.

The lottery puzzle

To add a block, a miner must find a nonce that makes the block’s hash start with a required number of zeros. Hashes are unpredictable, so there’s no shortcut: miners simply try nonce 0, 1, 2, 3… billions of times per second. Finding a winning nonce is hard; checking it takes anyone a single hash. That asymmetry is the magic.

  • Difficulty — each extra required zero multiplies the expected work by 16 (in hex). Networks auto-tune difficulty to keep block times steady.
  • Reward — the winning miner earns new coins plus fees. This is how new cryptocurrency enters circulation.
  • Security — rewriting history means redoing all that work and outpacing every honest miner. The energy cost is the lock on the vault.

Built-in scarcity: the halving

Mining doubles as Bitcoin’s mint: the block reward is the only way new coins are ever created. And the mint runs on a countdown — every 210,000 blocks (roughly four years) the reward halves: 50 BTC at launch, 25 from 2012, 12.5 from 2016, and so on. The halvings turn issuance into a geometric series whose sum stops just short of 21 million BTC, with the last fraction of a coin mined around the year 2140.

The difficulty thermostat

If mining is a lottery, what stops blocks arriving faster as more miners join? Every 2,016 blocks (about two weeks) Bitcoin measures how long they actually took. Faster than the 10-minute target → difficulty rises proportionally; slower → it falls (clamped to a 4× change per adjustment). The result is a thermostat: hashpower can grow a millionfold — and it has — yet blocks keep their ~10-minute heartbeat, and coins are issued no faster. Extra hashpower buys miners a bigger share of the same rewards, while the network simply gets more expensive to attack.

The thermostat works in reverse too. When China banned mining in mid-2021, roughly half the world’s hashpower vanished within weeks; blocks crawled, and the next adjustment dropped difficulty ~28% — the largest cut ever — restoring the rhythm. (Proof-of-stake chains skip all this: Ethereum proposes a block every fixed 12-second slot, no race involved.)

From laptop CPUs to warehouse ASICs — and pools

Mining hardware escalated in four waves: CPUs (2009, Satoshi and a handful of hobbyists), GPUs (2010, ~100× faster at parallel hashing), FPGAs (2011, briefly), then ASICs (2013 on) — chips that can do nothing except SHA-256, but do it millions of times faster per watt than a CPU, and are obsolete within a couple of years. Solo mining became a lottery ticket with terrible odds, so miners formed pools: thousands combine hashpower and split each reward in proportion to work contributed, trading jackpot odds for a steady trickle.

Pools create a real centralization tension: the pool operator chooses which transactions go in the blocks, and a handful of large pools often add up to more than half the network. The counterweight is that pool members can switch pools in minutes — operators who misbehave bleed hashpower — but it’s a live debate, not a solved problem.

The energy question, honestly

Proof-of-work security is energy — that’s the design, not a side effect. Estimates put Bitcoin around 100–150 TWh per year, roughly half a percent of world electricity, comparable to a mid-size country. Defenders point out that miners chase the cheapest power and so soak up stranded hydro, flared gas and off-peak surplus; critics answer that the footprint scales with the coin’s price, not its usefulness, and that ASIC churn adds e-waste.

One engineering fact both sides accept: proof of stake delivers comparable security guarantees from economic collateral instead of electricity — Ethereum’s 2022 switch (“the Merge”) cut its energy use by ~99.95%. Bitcoin keeps proof of work deliberately: its community regards a security budget rooted in physics, outside the system itself, as worth the cost. That disagreement — physical cost versus financial collateral — is the real content of the energy debate, and Module 6 compares the two models head-on.

Module 5

Keys, wallets & signatures

If hashes protect the history, digital signatures protect the ownership. Your “account” on a blockchain is nothing more than a key pair.

The key pair

  • Private key — a giant random number only you know. It is your identity. Whoever holds it controls the funds — there is no “forgot password” link.
  • Public key — derived from the private key and safe to share. Your address is (roughly) a hash of it.
  • Signature — a proof, computed with your private key, that you approved a specific message. Anyone can verify it with your public key; nobody can forge it without your private key.

A wallet doesn’t hold coins — coins only exist as ledger entries. A wallet holds your keys and signs transactions on your behalf.

Custody in one sentence

“Not your keys, not your coins” — if an exchange holds your private keys, you hold an IOU, not cryptocurrency. Hardware wallets and seed phrases exist purely to keep private keys safe and recoverable.

The math: elliptic curves in plain words

Your private key is just a 256-bit number, k. Your public key is the result of starting from a fixed point G on a special curve and “hopping” along it k times — written k × G. Computing the hops forward is fast (a few hundred operations, using doubling tricks); going backward — recovering k from the public key, the “discrete logarithm problem” — has no known shortcut. The only attack is brute force through a 2²⁵⁶ keyspace, and Module 2’s deep dive showed how that ends.

Bitcoin and Ethereum both use the curve secp256k1. The demo above uses P-256, a sibling curve, because that’s the one your browser’s Web Crypto API ships — same construction, different constants. One honest caveat: a large-scale quantum computer running Shor’s algorithm would break both. Post-quantum signature schemes already exist and chains can migrate to them; it’s a watched-but-distant problem, and coins marketed today as “quantum-proof” are mostly marketing.

Seed phrases & HD wallets: one secret, endless keys

Nobody should hand-copy a 256-bit number, so wallets encode their master randomness as 12 or 24 words drawn from a fixed 2,048-word list (the BIP-39 standard) — words are far harder to mistranscribe than hex. From that seed, a hierarchical deterministic (HD, BIP-32) wallet derives an unlimited tree of key pairs along well-known paths, fresh address per payment included. Restore the phrase on a brand-new device and every key — every coin — reappears.

That’s the magic and the danger in one sentence: the phrase is the wallet. Anyone who reads it owns everything derived from it, instantly and irrevocably. Hence the rules: write it on paper or steel, never in a photo, cloud note or password manager synced online; and treat anyone asking for it — “support staff” above all — as an attacker by definition. Many wallets add an optional passphrase (“25th word”) so the paper backup alone isn’t sufficient.

From public key to address

An address is not the public key — it’s a hash of it, shortened and wrapped in a typo-resistant encoding. Bitcoin runs the public key through SHA-256 then RIPEMD-160 and encodes the result (legacy 1… and 3… formats, modern bc1… bech32); Ethereum takes the last 20 bytes of the Keccak-256 hash, giving the familiar 0x…. The encodings carry checksums, so a mistyped address is rejected by your wallet instead of sending coins into the void — one of the few mercies in this system. Why hash at all? It’s shorter, and it keeps the public key itself off-chain until the first time you spend.

How coins actually get stolen

Essentially never by breaking the cryptography — the math holds. Real losses come from the seams around it:

  • Phishing — fake wallet sites, fake airdrops, “support” DMs asking for your seed phrase.
  • Malware — clipboard swappers that silently replace the address you just copied with the thief’s.
  • SIM swaps — hijacking your phone number to defeat SMS-based logins at exchanges.
  • Malicious approvals — signing a contract interaction that grants unlimited spending of a token (Module 7).
  • Custodial failure — exchanges collapsing with customer funds (Mt. Gox 2014, FTX 2022): the “not your keys” lesson, at scale.

The defenses are unglamorous: a hardware wallet for savings, verifying addresses character-by-character (or via a second channel), keeping the seed offline, and assuming any request for it is a scam — because it is.

Module 6

Transactions & consensus

You now know every ingredient. Let’s follow a transaction end to end, then look at the two big ways networks agree on the next block.

Life of a transaction

  1. Create — your wallet drafts “Alice pays Bob 5”, referencing coins you own.
  2. Sign — your private key signs it (Module 5).
  3. Broadcast — it spreads node-to-node into the mempool, the waiting room of unconfirmed transactions.
  4. Confirm — a miner/validator packs it into a block (Modules 3–4). Each later block buries it deeper; ~6 confirmations is considered settled on Bitcoin.

Consensus rules also solve the double-spend problem: if you broadcast two transactions spending the same coin, the network will only ever confirm one — the other is rejected as invalid.

Beyond currency

Once a network can agree on arbitrary data, it can agree on code. Smart contracts are programs stored on-chain that execute exactly as written — powering tokens, decentralized exchanges, lending markets (DeFi) and NFTs. Different goals, same foundations you just learned — and exactly where Module 7 goes next.

Two ways to track money: UTXO vs accounts

Bitcoin has no balances anywhere in its data. It tracks UTXOs — unspent transaction outputs — which behave like bills in a cash drawer: a transaction consumes whole bills and mints new ones, including change back to yourself. Pay 3 BTC from a 5-BTC output and the transaction creates two outputs: 3 to the merchant, ~2 back to you (minus the fee). Your wallet’s “balance” is just it adding up every bill you can still spend.

Ethereum instead keeps an account model: real balances, debited and credited, plus a per-account counter (the nonce — same word as mining’s nonce, different job) that orders your transactions and blocks replays. The trade-offs: UTXOs parallelize well and make coin-tracing slightly harder; accounts are far simpler for smart contracts, which need persistent state — you wouldn’t want to rebuild a token’s balance table out of bills on every call.

Soft forks, hard forks and chain splits

Who decides the rules of a system with no boss? Rule changes come in two flavors. A soft fork tightens the rules: blocks valid under the new rules are still valid under the old, so un-upgraded nodes keep following along (Bitcoin’s SegWit 2017 and Taproot 2021 shipped this way). A hard fork changes rules incompatibly: old nodes reject the new blocks, and if part of the community keeps mining the old rules, the network splits into two chains, each carrying the full shared history — which is why holders of pre-fork coins end up with coins on both.

The famous splits were governance disputes settled by exit: Bitcoin Cash left Bitcoin in 2017 over block sizes, and Ethereum Classic is the chain that refused the 2016 DAO bailout fork (the full story is a Module 7 deep dive). Forks are blockchain politics: when consensus about the rules fails, the ledger itself divides.

How final is “final”?

Proof-of-work finality is only ever probabilistic: every confirmation buries your transaction deeper and makes reversal exponentially costlier, but the probability never hits zero. On giant networks that’s academic; on small ones it isn’t — Ethereum Classic and Bitcoin Gold both suffered real 51% double-spend attacks (2018–2020) because enough hashpower to overwhelm them could be rented by the hour. Security budgets matter.

Ethereum’s proof of stake adds explicit checkpoints: validators vote on recent history, and once two-thirds of all staked ETH attests, the checkpoint is finalized (~13 minutes). Reverting a finalized block isn’t “very expensive” — the protocol guarantees it would destroy at least one-third of all staked ETH, tens of billions of dollars burned. The attack cost shifts from out-computing the network to incinerating your own capital.

Beyond PoW and PoS

Consensus is a design space, and every point in it is a different answer to Module 1’s trilemma:

  • Delegated PoS — coin holders elect a small validator set (EOS, Tron). Fast and cheap; visibly more centralized.
  • BFT committees — a bounded validator set votes on every block (Tendermint/Cosmos). Instant, clean finality; the validator list is the trust assumption.
  • Proof of authority — named, legally identifiable validators. Fine for testnets and consortium chains; not trustless at all.
  • Hybrids — e.g. Solana pairs PoS with “proof of history”, a cryptographic clock that lets validators order events without waiting on each other, buying huge throughput at the cost of heavyweight hardware requirements.

Module 7

Ethereum & smart contracts

Bitcoin proved a network can agree on money. Ethereum asked: why stop there? If thousands of nodes can agree on a ledger, they can agree on the result of running a program.

One idea, two chains

Ethereum (launched 2015) keeps everything you’ve learned — blocks, hashes, keys, consensus — but turns the ledger into a world computer: alongside balances, it stores code and that code’s data. Every node runs the Ethereum Virtual Machine (EVM), so a program executes identically everywhere and the network agrees on its results the same way it agrees on payments.

Two kinds of accounts

Externally owned account
Controlled by a private key — the wallets you built in Module 5. It can hold ETH and send transactions.
Contract account
Controlled by code, not a key. It has its own address and balance, and reacts only when a transaction calls it.
Smart contract
The program inside a contract account. Once deployed it runs exactly as written — its author can’t reach in and bend the rules.
Token
Not a separate blockchain — just a contract keeping a balance table. ERC-20 (currencies) and ERC-721 (NFTs) are standard shapes for that table.

Try it: call a smart contract

Below is a tiny token contract. Its one rule lives in the require(...) line: you can’t send tokens you don’t have. Pick a caller and try to break it. 

contract LabToken {
  mapping(address => uint) balances;

  function transfer(address to, uint amount) {
    require(balances[msg.sender] >= amount, "insufficient balance");
    balances[msg.sender] -= amount;
    balances[to] += amount;
    emit Transfer(msg.sender, to, amount);
  }
}

Gas: paying the world computer

Running code on thousands of machines isn’t free. Every EVM operation has a price measured in gas; you pay gas used × gas price, in ETH. Gas does two jobs: it pays validators for computation and storage, and it makes spam and infinite loops impossible — a program that runs out of fuel simply stops and its transaction reverts.

  • Gas used — depends on the work: a plain ETH transfer is exactly 21,000 gas; calling complex contracts costs more.
  • Gas price — quoted in gwei (a billionth of an ETH) and set by supply and demand for block space. Busy network → higher price.
  • Layer 2 rollups — networks that batch thousands of transactions off-chain and post a compressed proof to Ethereum, cutting fees 10–100×.

What people build with this

Once contracts can hold money and enforce rules, finance becomes software: tokens (currencies and loyalty points as contracts), DeFi (exchanges and lending markets with no company in the middle), NFTs (contracts tracking ownership of unique items), and DAOs (organizations whose treasury and voting rules are code). Every one of them is just a program — like LabToken, only bigger.

Go deeper

The modern fee market: EIP-1559

Since August 2021 the “gas price” you saw in the estimator is really two parts: fee = gas used × (base fee + tip). The base fee is set by the protocol itself, block by block: if a block runs more than half full, the base fee rises 12.5% for the next one; less than half full, it falls 12.5%. Congestion pricing with no auction — wallets can finally quote a fee instead of guessing a bid.

The twist: the base fee is burned — destroyed, paid to no one. Only the small optional tip goes to the validator. Burning kills the incentive for validators to congest the network to drive fees up, and it offsets new ETH issuance; in busy periods Ethereum has been net-deflationary. When you read a gas tracker, mentally split the number: a big burned base fee plus a sliver of tip.

When “code is law” goes wrong: The DAO

In 2016 a crowdfunded, code-governed venture fund called The DAO raised about $150M in ETH — then an attacker found a reentrancy bug: its withdraw function sent ETH before updating the caller’s balance, and the recipient’s code could call back into withdraw again, and again, draining ~3.6M ETH through a perfectly legal sequence of calls. The contract did exactly what it was written to do; it just wasn’t written to do what its authors meant.

The community faced the philosophy head-on: accept the theft (“code is law”) or hard-fork history to unwind it. The majority forked — that chain is the Ethereum you know — while the minority kept the untouched ledger as Ethereum Classic. The episode built the modern safety culture: audits, bug bounties, formal verification, and patterns like checks-effects-interactions — notice LabToken above checks require(...) and updates balances before any interaction. Immutability cuts both ways: unstoppable rules, unstoppable bugs.

Oracles: how contracts learn about the world

The EVM is deterministic and sealed: a contract cannot fetch a URL, because thousands of nodes would get different answers at different times and consensus would shatter. Any off-chain fact — a price, a match result, the weather — must be written in by a transaction. The service doing the writing is an oracle, and it quietly reintroduces trust: your trustless contract is exactly as honest as its data feed. This is the oracle problem.

The standard mitigation is decentralizing the messenger — networks like Chainlink aggregate many independent operators and data sources, with staked collateral to slash liars. It works well enough to carry billions, but oracle manipulation (often pumping a thin market with a flash loan so a lending contract mis-prices collateral) remains one of the largest categories of DeFi exploit. When you evaluate any DeFi protocol, the first question is: where does its price come from?

Stablecoins: the dollar, tokenized

The most-used tokens aren’t volatile coins — they’re stablecoins, engineered to track the dollar. Three designs, three trust models:

  • Fiat-backed (USDC, USDT) — a company holds dollars and T-bills, and mints one token per dollar. Simple and dominant, but custodial: you’re trusting the issuer’s reserves, and the issuer can freeze addresses.
  • Overcollateralized crypto (DAI) — lock ~$150+ of ETH in a contract to mint $100 of stablecoin; automatic liquidations keep the system solvent. Trustless-ish, capital-inefficient.
  • Algorithmic — peg maintained by mint-and-burn incentives with no full collateral. Terra/UST collapsed this way in May 2022, erasing ~$40B in days. Treat the category as a cautionary tale.

Stablecoins are arguably crypto’s most adopted product: most on-chain trading is denominated in them, and they move serious remittance volume. They’re also where blockchains and bank regulators meet.

Rollups, properly: optimistic vs ZK

Both kinds of layer 2 execute transactions off-chain and post compressed data back to Ethereum; they differ in how Ethereum knows the results are honest. Optimistic rollups (Arbitrum, Optimism, Base) assume batches are valid and give challengers ~7 days to submit a fraud proof showing a bad state transition — secure, but native withdrawals wait out the window. ZK rollups (zkSync, Starknet, Scroll) submit a validity proof — a succinct cryptographic certificate that the batch was executed correctly — which an L1 contract verifies in milliseconds before accepting; no challenge window, at the cost of much heavier math to generate.

Either way the rollup inherits Ethereum’s security rather than bootstrapping its own — that’s the trilemma answer promised in Module 1. The economics improved sharply in March 2024 when EIP-4844 gave rollups “blobs”: cheap, short-lived data space on L1 that dropped typical L2 fees to cents. The roadmap bet is that users live on L2s while Ethereum’s base layer becomes the settlement court.

Glossary

Quick definitions for the terms used across the course. One opens at a time.

Blockchain

A shared, append-only ledger replicated across many computers, where blocks of records are linked by hashes so history can’t be quietly rewritten.

Block

A batch of transactions plus metadata: the previous block’s hash, a nonce, and its own hash. The first block of a chain is the genesis block.

Hash

A fixed-size fingerprint of any data, produced by a one-way function such as SHA-256. Any change to the input produces a completely different hash.

Merkle tree / root

A tree of hashes: transactions are hashed in pairs, then the pairs are hashed in pairs, up to a single Merkle root stored in the block header. Changing any transaction changes the root, and inclusion of one transaction can be proven with only ~log₂(n) hashes.

Node

A computer that stores the ledger, validates new blocks and relays transactions. Anyone can run one.

Nonce

“Number used once” — the value miners vary to search for a hash that satisfies the difficulty target.

Mining

Racing to find a valid nonce for the next block. The winner earns newly issued coins plus transaction fees.

Mining pool

Miners who combine hashpower and split each block reward in proportion to work contributed — trading lottery-style jackpots for a steady trickle.

Difficulty adjustment

The automatic re-tuning of the proof-of-work target (every 2,016 blocks on Bitcoin) so blocks keep arriving roughly every 10 minutes no matter how much hashpower joins or leaves.

Halving

Bitcoin’s block reward cuts in half every 210,000 blocks (~4 years): 50 → 25 → 12.5 → … BTC. It’s why total supply approaches, but never exceeds, 21 million coins.

Proof of Work (PoW)

Consensus by computation: blocks are expensive to produce but cheap to verify, so rewriting history costs more than honest participation.

Proof of Stake (PoS)

Consensus by collateral: validators lock up coins for the right to propose blocks, and lose them (“slashing”) if they cheat.

Private key

A secret number that controls an address’s funds. Whoever knows it owns the coins; lose it and the coins are gone forever.

Seed phrase

12 or 24 dictionary words (BIP-39) encoding a wallet’s master secret. An HD wallet re-derives every key from it, so anyone holding the words holds all the coins.

Public key / address

The shareable half of a key pair. An address — what you give people to get paid — is essentially a hash of the public key.

Digital signature

Proof, verifiable by anyone, that the holder of a private key approved one exact message. The basis of transaction authorization.

Wallet

Software or hardware that stores keys and signs transactions. It holds no coins — coins exist only as ledger entries.

Cold / hot wallet

Cold = keys kept on a device that never touches the internet (hardware wallet, steel plate); hot = keys on an online device. Convenience goes up, safety goes down.

Mempool

Each node’s waiting room of valid but not-yet-confirmed transactions, from which miners/validators assemble blocks.

Confirmation

Each block added after the one containing your transaction. More confirmations = deeper burial = harder to reverse.

Finality

The point after which a transaction can’t be reversed. Probabilistic under PoW (deeper = safer, never absolute); explicit under Ethereum PoS, where reverting a finalized block requires destroying at least ⅓ of all staked ETH.

Double-spend

Trying to spend the same coin twice. Consensus guarantees at most one of the conflicting transactions is ever confirmed.

UTXO

Unspent transaction output — Bitcoin’s version of a coin, behaving like a bill: transactions consume whole UTXOs and create new ones, including “change” back to the sender. Ethereum uses account balances instead.

51% attack

Controlling a majority of mining power (PoW) or stake (PoS), enough to censor or reorder recent transactions. Prohibitively expensive on large networks.

Fork (soft / hard)

A change to the protocol rules. Soft forks tighten rules and stay backward-compatible; hard forks don’t — if part of the network refuses to upgrade, the chain splits in two (Bitcoin / Bitcoin Cash, Ethereum / Ethereum Classic).

Blockchain trilemma

The rule of thumb that decentralization, security and scalability trade off against each other — base layers usually maximize the first two and reach for scale via layer 2.

Smart contract

A program stored on-chain that runs exactly as written when triggered by a transaction — the engine behind tokens, DeFi and NFTs.

Oracle

A service that feeds off-chain facts (like prices) to smart contracts, which cannot fetch external data themselves. A trustless contract is only as honest as its oracle — the “oracle problem”.

Ethereum / EVM

The largest smart-contract blockchain. Every node runs the Ethereum Virtual Machine (EVM), so contracts execute identically everywhere; ETH is its native coin and pays for gas.

Gas / fees

The price paid to have the network process your transaction or run your contract; it compensates miners/validators and prevents spam.

Gwei

A billionth of an ETH — the unit gas prices are quoted in. Your fee is gas used × gas price in gwei.

Base fee & tip (EIP-1559)

Ethereum’s fee model since 2021: a protocol-set base fee that is burned (rising 12.5% when blocks run over half full, falling when they don’t) plus an optional tip that pays the validator.

Satoshi / wei

The smallest units of the major coins: 1 BTC = 100,000,000 satoshis; 1 ETH = 10¹⁸ wei. On-chain amounts are integers of these units — the decimals are just display.

Token vs. coin

A coin is a blockchain’s native currency (BTC, ETH). A token is an asset created by a smart contract on top of an existing chain.

Stablecoin

A token engineered to track a currency like the US dollar — via custodial reserves (USDC, USDT), overcollateralized crypto (DAI), or — riskily, see Terra 2022 — pure algorithms.

Layer 2 / rollup

A network that executes transactions cheaply off the main chain, then posts compressed proof of them on-chain — inheriting its security at a fraction of the fee.

DeFi & NFT

Decentralized finance: lending, trading and earning interest via smart contracts. NFTs: unique on-chain tokens representing ownership of a specific item.

DAO

Decentralized autonomous organization — members coordinate through tokens and on-chain votes, and the treasury is controlled by a smart contract instead of officers. (“The DAO”, 2016, was the first and most infamous one.)
